# 《独家童话》第四阶段：UI 验收与体验问题台账

> 状态：**待启动的体验改进台账，不是当前第二阶段的阻塞项。**
>
> 本文记录第一、二、三阶段测试中已发现、但不影响当前真实数据闭环验证继续进行的问题。第四阶段预计会与第一阶段 UI 验收和渐进式美化工作合并排期；不得据此扩展第二阶段业务或宣称问题已经修复。

## 1. 阶段定位与启动条件

第四阶段的目标是把已完成的业务页面提升到可验收的产品体验，而不是新增业务范围。它覆盖：

- 第一阶段 UI 视觉、交互、加载与空状态的统一验收；
- 第二阶段真实数据链路在真实网络下暴露的非阻塞体验问题；
- 第三阶段接入前必须明确的认证入口与能力设计债务。

本阶段暂不启动。启动前应至少满足：

1. 第二阶段的 Supabase 迁移、A/B/C 权限验证和双账号 Web 闭环有可追溯结果；
2. 第一阶段完成新的全页面 UI 验收清单，明确保留、返工和待设计页面；
3. 第三阶段 AI 计划仍保持独立，不把 AI、支付或真实短信接入混入本阶段。

## 2. 问题台账

### P4-01：首次进入加载时间过长

**状态**：**已解决（2026-07-18，Web Real 模式）**。

**原始现象**：Real 模式下，登录进入或刷新后，“正在打开你们的童话…”加载层可能停留十秒以上。

**根因与处理**：

- 根布局把约 10 MB 的自定义字体作为 Web 路由挂载的前置条件；字体请求未完成时，整个应用一直显示启动层。
- 启动流程在离开全屏启动层前串行等待情侣关系以外的日记、相册、纪念日、标签、胶囊和通知请求；任一模块慢或失败都可能拖慢首屏。
- Web 首屏改用系统字体，不再等待自定义字体；移动端仍保留原有字体注册。
- 启动阶段只等待已验证 Session 与当前情侣关系（关系请求 5 秒上限）；业务模块改为后台加载，并使用独立的 `modules` 加载状态。
- 登录校验增加整体 5 秒上限，远程身份确认限制为 4 秒；网络异常会结束启动检查并进入既有登录/错误路由，而不会无限停留在加载层。

**已验证证据**：定向 Jest（16 项）与页面、Real 能力、Real 页面、API 四项静态审计均已通过；已连接的真实 Chrome 登录会话在更新后直接渲染首页，未出现“正在准备/打开你们的童话…”启动层。

**保留边界**：这只解决了首屏被无关字体和非关键模块长时间阻塞的问题；P4-02 的加载层视觉设计、Android 真机冷启动量测，以及弱网基线仍按各自计划执行。

**第四阶段处理范围**：

- 在不记录令牌、密码或隐私内容的前提下，为认证、关系加载、各业务模块加载和 Realtime 建连添加可关闭的耗时观测；
- 区分首屏必要数据（Session、当前关系、首页最小展示信息）和可在首屏后加载的数据（日记、相册、标签、胶囊、通知等）；
- 审计 `INITIAL_SESSION`、登录事件、Token 刷新和 Realtime 刷新是否重复触发 bootstrap；
- 设计请求去重、取消、超时、失败重试和缓存策略，确保不会跨账号展示旧缓存；
- 在 Web 与 Android 真机的冷启动、热启动、弱网和断网场景分别记录证据。

**验收标准**：先记录基线，再为目标网络与设备定义可量化的首屏时间预算；优化后必须证明真实数据、Session 隔离、RLS 边界和失败提示均未被“为了更快”而跳过。

### P4-02：“正在打开你们的童话…”加载层视觉与交互不足

**现象**：当前加载层功能正确但视觉过于单调，不能达到第一阶段的童话绘本 UI 验收标准。

**第四阶段处理范围**：

- 以 `FairyPage` 为骨架设计全屏加载体验，延续奶油纸张、手绘贴纸和品牌插画语言；
- 使用轻量、可降级的进度/状态提示；不伪造百分比、业务成功或伴侣数据；
- 完整覆盖初始加载、较慢网络、超时、失败、重试、Session 失效和无障碍减少动态效果；
- Web 与 Android 验证 Safe Area、屏幕旋转、键盘和低性能设备表现；
- 复用现有主题 token 与素材治理规范，新增素材必须有来源、命名和注册记录。

**验收标准**：视觉方案通过第一阶段页面级验收；加载、错误与重试可操作；不增加阻塞动画、不泄露账户信息，也不掩盖实际网络失败。

### P4-03：手机号注册/登录与验证码方案

**现象**：当前仅有邮箱密码注册和登录。后续可能需要手机号作为注册和登录入口，但尚未进行产品、安全、成本和 API 设计。

**当前边界**：本项只是设计与评审事项，第四阶段之前不得接入真实短信供应商、验证码密钥、生产短信费用或客户端密钥。

**第四阶段设计要求**：

- 明确产品选择：手机号登录、手机号注册、邮箱绑定手机号，或以上组合；定义与现有邮箱账户的唯一身份、合并和冲突规则；
- 明确支持地区、号码格式、国际区号 UI、隐私提示、服务条款与短信成本上限；
- 客户端只调用统一 `authApi` 接口，例如“请求验证码、校验验证码、绑定/解绑号码”，页面不得直接调用供应商 SDK；
- 服务端/认证提供方负责发送验证码、校验、频率限制、IP/设备风控、验证码有效期和一次性消费；客户端不得保存验证码、服务端密钥或供应商 Secret；
- 评估 Supabase Phone Auth 与可替换短信供应商的适配层，确认区域可用性、模板、费用、回调、测试号码和失败降级；
- 设计防刷：同号码、同 IP、同设备、同会话的速率限制，图形/行为挑战策略，以及审计日志的最小化保留；
- 覆盖手机号已存在、邮箱已存在、验证码错误/过期、发送频繁、号码变更、注销、Session 恢复和账号合并的 RLS 与 UI 测试。

**验收标准**：先完成无密钥的接口契约、Mock/Real Capability、威胁模型和测试矩阵；只有在用户确认供应商、地区、预算和合规要求后，才允许执行真实短信联调。

### P4-04：FairyPage 的 Web 容器宽度与响应式展示规范

**现象**：当前页面已普遍以 `FairyPage` 作为骨架，但在桌面浏览器中内容会横向撑满整个视口，缺少与移动端产品定位一致的最大阅读宽度、边距和背景层次。

**第四阶段处理范围**：

- 在 `FairyPage` 或其明确的 Web 容器层集中定义响应式约束；不得在各业务页面分散硬编码不同宽度；
- 为 Web 端设定可配置的内容最大宽度、最小安全边距和居中策略；大屏展示应用背景，内容仍保持适合手机界面的阅读与操作密度；
- 保证窄屏、平板、桌面、横屏以及 Safe Area 下布局连续，不裁切 `FairyHeader`、底部操作区、弹窗或键盘交互；
- 审查 `FairyPage` 覆盖的所有实际业务页，并对不使用该骨架的登录、重定向和开发工具页面明确例外规则；
- 不以固定像素截图作为唯一验收依据，同时进行 Web 实机多宽度检查和 Android 真机回归。

**验收标准**：Web 内容不会在常见桌面窗口中无边界拉伸；核心页面在约 360px、768px、1280px 及更宽视口均保持一致的品牌层次和可操作性；移动端现有布局不回归。

### P4-05：双账号 Web 联调的浏览器会话隔离

**现象**：同一浏览器 Profile 的同一站点共享 Cookie、LocalStorage 和 Supabase Session，因此不能在两个普通标签页中同时维持 A、B 两个不同账号登录；后打开的登录会覆盖先前会话。

**当前边界**：这不是情侣关系、RLS 或邀请码业务缺陷，也不应通过在客户端保存两份业务 Session 来规避。生产产品仍应遵循单个浏览器 Profile 对同一站点只保留一个活动账号的安全预期。

**第四阶段处理范围**：

- 固化双账号 Web 验收方式：使用两个独立浏览器 Profile、普通窗口与无痕窗口，或两种独立浏览器；每个会话只登录一个测试账号；
- 为测试步骤补充“当前账号、Profile/会话、预期数据归属”记录，避免把缓存或 Realtime 更新误判为跨账号泄漏；
- 审查登出、账号切换和 Session 失效后的 Store、AsyncStorage 命名空间及 Realtime Channel 清理，确保同一会话切换账号时没有旧数据残留；
- 如未来需要运营/客服代入或多身份并行操作，另行设计受控的模拟身份工具，不复用真实用户双 Session。

**验收标准**：A/B 双账号闭环可在隔离会话中稳定复验；同一会话登录切换后不会保留前一账号的页面缓存、订阅或业务数据；不要求同一 Profile 的两个普通标签页同时登录两个账号。

### P4-06：长页面中的系统反馈可见性与错误提示位置

**现象**：当前部分页面依赖底部 `FairyToast` 呈现权限、网络或提交失败信息。页面内容较长、用户正在顶部编辑或键盘打开时，提示可能落在可视区域之外；例如 `PERMISSION_DENIED` 被规范化为“当前账号没有权限执行此操作”，用户却无法立即看到原因。

**第四阶段处理范围**：

- 建立统一的反馈呈现规则：表单提交、上传、删除和权限失败必须显示在触发操作附近、页面顶部固定反馈区或安全区内的浮层；不得只依赖长页面底部位置。
- 保留 API 的结构化错误契约，例如 `error.message`、`error.code = 'PERMISSION_DENIED'` 和 `error.category = 'permission'`；页面只消费友好文案，不暴露原始 Supabase 错误、RLS 表达式、令牌或路径。
- 对权限、Session 失效、网络可重试、冲突和校验失败分别定义可见文案、重试/重新登录/修改输入动作，以及提交按钮的恢复状态。
- 审查 `FairyToast`、`FairyRequestState`、`FairyPage`、键盘避让和 Safe Area 的协作方式；在 Web、Android 小屏和长表单场景保证提示不会被底部导航、键盘或滚动内容遮挡。
- 对所有真实写入页面建立回归清单：日记、照片、纪念日、标签、胶囊、评论、情侣资料和通知已读；同类错误不允许各页自行决定不同位置或吞掉错误。

**验收标准**：在长页面顶部、底部和键盘打开状态下，构造权限拒绝（如 `expect(response.error).toMatchObject({ message: '当前账号没有权限执行此操作', code: 'PERMISSION_DENIED', category: 'permission' })`）时，用户无需滚动即可看到明确反馈和下一步动作；错误不泄露内部实现细节，成功与失败均不会造成重复提交或页面状态错乱。

### P4-07：私有图片上传性能、并发与过程反馈

**现象**：第二阶段 A/B 实际联调已确认：A 上传照片并跳转展示页后，B 可通过 Realtime 自动更新照片数量，说明 Storage、数据库写入和实时同步闭环正确；但 A 的上传等待时间偏长。当前链路会读取完整本地文件、校验会话和情侣关系、上传至私有 Bucket、写入照片集与照片记录、生成签名 URL。多图使用串行 `for...of` 上传，最多 9 张图片会依次等待，Web 端原始大图的读取和上传时间尤其明显。

**当前边界**：这是体验与性能债务，不以取消 Storage RLS、跳过服务端成功确认、提前伪造保存成功或把私有路径改为公开 URL 作为优化手段。第二阶段已完成的对象路径 `{coupleId}/{userId}/{uuid}`、作者删除和失败补偿约束必须保持。

**第四阶段处理范围**：

- 建立上传耗时观测：分别记录本地文件读取/压缩、单文件上传、照片集写入、照片记录写入、签名 URL 生成和页面跳转耗时；仅记录大小、数量、耗时和错误分类，不记录图片内容、Token 或签名 URL。
- 为 Web 与 Android 设计统一的图片预处理策略：像素尺寸上限、目标质量、格式兼容、单图大小上限和无法压缩时的友好提示；保留原始照片的产品规则需另行确认，不能默认牺牲用户照片质量。
- 上传动作复用本次操作已获取的认证/情侣上下文，避免为同一批照片重复请求会话与关系；不得复用到登出、切换账号或情侣关系变化后的操作。
- 将多图上传从全串行调整为受控并发（例如 2–3 张），保留批次级取消、失败停止、已上传对象清理和失败补偿元数据；禁止无上限并发导致弱网、内存或 Storage 限流问题。
- 设计可见的过程反馈：显示总数量、当前阶段或已完成数量、可取消/重试状态和失败图片归属；与 P4-06 的顶部或安全区反馈规则一致，不能只在长页面底部提示“正在保存”。
- 在弱网、大图、单图、多图、部分失败、Session 失效、权限拒绝、页面离开和重新进入场景回归；确认 B 的 Realtime 更新只在 A 服务端写入成功后触发，不因预处理或中途上传而提前增加数量。

**验收标准**：在约定网络和设备基线下，单图及多图上传的端到端耗时有可比较的优化前后证据；进度可见且不会卡死；失败时未产生孤儿 Storage 对象或错误照片记录；A 成功后 B 才更新，C 仍无法读写；Web 与 Android 的图片大小、质量和失败提示符合统一产品规则。

### P4-08：情侣共享内容的作者权限可见性与协作策略

**现象**：第二阶段 A/B 联调确认：B 能看到 A 上传的私有照片，且当 B 尝试删除时服务端正确返回“照片集不存在、无权限或已被删除”。但相册界面仍向非上传者显示“移除记录”按钮，用户点击后才得知没有权限，造成无效操作和误解。

**第四阶段处理范围**：

- 对日记、照片集/照片、评论、资料、标签、纪念日和时光胶囊逐项明确“作者独占”“情侣共同维护”“仅本人可见”的产品归属；不能只依据当前 RLS 行为推断 UI 权限。
- 在已确定作者独占的内容上，页面只对作者展示编辑/删除入口；非作者保留查看、评论或其他已授权操作，不展示必然失败的 destructive action。
- 如果产品决定某类内容支持情侣共同维护，须先新增受控服务端授权、审计字段、冲突处理和通知规则，再展示共同编辑/删除入口；不得仅放宽客户端 UI 或 RLS。
- 删除入口隐藏不是安全边界：服务端 RLS、API 作者校验和 Storage 对象所有者校验必须继续保留，并以 A/B/C 回归证明隐藏 UI 后越权请求仍被拒绝。

**验收标准**：每项共享内容的 UI 操作入口与服务端权限矩阵一致；非作者不会看到必然失败的删除/编辑操作；作者权限仍可正常执行；C 账号始终无权访问；共同维护能力只有在完整服务端规则和冲突策略上线后才开放。

## 3. 执行顺序

1. 先完成第一阶段全页面 UI 验收与问题分级；
2. 以 P4-01 的性能基线决定首屏加载改造边界；
3. 在确定加载状态模型后实施 P4-02 的视觉与交互设计；
4. P4-03 先完成产品/安全/API 设计评审，真实短信接入另行授权；
5. P4-04 先形成统一容器规范，再按页面清单完成 Web 与移动端回归；
6. P4-05 先建立隔离会话的双账号验收环境，再复验缓存、登出和 Realtime 边界；
7. P4-06 在各真实写入页的 UI 验收批次中统一实施，先定义全局反馈组件与位置规则，再逐页回归。
8. P4-07 在 P4-01 的性能基线和 P4-06 的反馈规范明确后实施；先完成单图和多图的测量，再决定压缩与受控并发参数。
9. P4-08 在各共享内容的产品归属规则确认后，先完成权限矩阵和服务端设计，再统一收敛可见操作入口。
10. 每个批次更新独立恢复检查点，并明确区分“代码通过”“Web 验证通过”“Android 真机通过”。

## 4. 非目标与记录规则

- 不把本台账中的问题标记为第二阶段失败，也不据此阻止当前 A/B 云端闭环测试；
- 不把 UI 导出、静态审计或 Mock 演示误报为真实网络性能验收；
- 不迁移 Mock 演示数据，不降低 RLS/Storage/RPC 约束，不为缩短加载而伪造业务成功；
- 后续新增非阻塞问题应按 `P4-xx` 编号追加：现象、影响、证据、建议范围、启动前置条件和验收标准。
