七、浏览器优先的开发方式

当前阶段不要求在开始开发前连接 Android 真机或模拟器。

默认开发流程：

1. 检查项目代码、Git 工作区和开发清单。
2. 找到目标页面的效果图、路由和实现文件。
3. 检查相关共享组件、主题和图片资源。
4. 完成页面代码和本地静态交互。
5. 检查当前 Expo Web 服务是否仍然可访问；可用时直接复用，失效时再重新启动。
6. 使用当前任务已加载的 `chrome:control-chrome` 技能连接用户 Chrome，打开或复用现有的 Expo Web 本地标签页。
7. 记录当前实际地址和端口。不将 `8081`、`8082` 或其他端口写死为唯一地址。
8. 在 Chrome 中检查布局、滚动、文字、图片、交互、响应式表现和可见的控制台问题。
9. 修复明显的视觉问题和运行错误。
10. 执行与当前改动风险匹配的构建或静态检查。
11. 完成本批页面并汇报。
12. 如果不存在阻塞，自动继续下一批页面。

可以使用以下方式提高浏览器调试效率：

- dev-ui-lab 页面隔离预览
- 开发环境专用页面入口
- 开发环境专用模拟数据
- 开发环境专用鉴权绕过
- Expo Web 构建检查
- 路由级页面检查

允许增加仅在开发环境中生效的入口、mock 或鉴权绕过，但必须满足：

1. 生产环境不生效。
2. 不删除正式鉴权逻辑。
3. 不改变真实业务入口。
4. 开发辅助代码具有清晰的环境判断。
5. 不把临时调试行为误带入生产环境。

如果 Chrome 控制暂时无法使用：

- 明确说明当前任务没有加载 Chrome 技能，或 Chrome 插件、扩展、标签页连接失效。
- 不得使用普通命令行打开浏览器、独立 Playwright 或其他方式冒充 Chrome 控制自测。
- 允许继续执行 Expo Web 构建和代码级跨平台检查，但必须将 Chrome 交互验证标记为未完成。
- 恢复 Chrome 连接后，再完成目标页面的 Chrome 自测。

不要因为没有连接模拟器或真机而停止页面开发。

Chrome 自测规则：

1. 优先复用本任务已启动的 Expo Web 服务和已打开的 Chrome 标签页。
2. 不要每处理一个页面就重启服务或新建重复标签页。
3. 服务失效时，检查原服务状态后重新启动，使用实际可用端口并记录。
4. 不终止用户原本已启动且与本任务无关的服务或 Chrome 标签页。
5. 自测至少包含页面标题、加载完成状态、目标路由、主要交互入口、滚动、返回、弹窗和输入操作。
6. 至少检查一个窄屏手机宽度和一个较宽 Web 窗口。
7. 控制台 error 必须解决后才能标记完成。
8. warning 需要判断来源；与当前改动有关的 warning 应修复，React Native Web 既有弃用或触摸 warning 可记录为已知问题，不得冒充 error，也不得完全忽略。

Expo Web 服务恢复边界：

1. 先对 `docs/resume-state.md` 中记录的地址和当前常用端口进行一次限时可访问性检查；能够返回页面时立即复用，不得继续规划重启。
2. 端口已被占用且页面可访问时，视为可复用服务；不要仅因无法确认进程归属而结束或重启该进程。
3. 端口已被占用但页面不可访问，且不能确认进程由当前任务启动时，不终止该进程，改用其他可用端口启动 Expo Web。
4. 服务启动或恢复最多尝试 2 次，每次从启动到可访问性判断最多等待 60 秒；不得无限等待 Metro 输出、端口释放、浏览器连接或进程状态变化。
5. 当前任务自己启动服务时，记录实际 URL、端口和可确认的进程信息到 `docs/resume-state.md`；不得把历史 PID 当作当前进程依据。
6. 两次尝试后仍无法恢复时，记录具体错误，将 Chrome 实时验证标记为待补，执行可运行的 Web 构建、静态检查和代码级跨平台检查，然后继续不依赖实时浏览器的页面代码、资源和文档工作。
7. 服务恢复失败不能导致任务长期停留在 Planning、等待或重复重启状态；只有目标页面的实现本身确实依赖实时运行结果、且没有安全替代工作时，才作为当前页面阻塞报告。
8. 一旦服务恢复，在同一执行轮次和后续批次持续复用；除非可访问性检查失败，不得主动重启。
9. 每批报告记录实际 URL、检查路由、主要交互结果、控制台 error 数量和 warning 概要。

九、Web 与移动端兼容是强制前提

本项目强制支持：

- Expo Web
- Android
- iOS

其中：

- Web 和 Android 是当前强制运行与功能兼容平台。
- Android 是最终重点设备验收平台。
- Web 是当前主要开发和调试平台。
- iOS 必须保持 React Native 通用代码、构建路径和核心功能兼容，但当前不要求 iOS 真机或模拟器视觉验收。

浏览器优先调试不代表只开发 Web 代码。所有新增页面、组件、交互、资源和依赖在编写时必须同时考虑 Web、Android 和 iOS。

功能兼容是强制要求；视觉允许根据平台特性有合理差异。

（一）共享代码优先

1. 默认使用同一套 React Native 代码支持 Web、Android 和 iOS。
2. 优先使用 View、Text、Image、Pressable、ScrollView、FlatList、TextInput、StyleSheet 和 useWindowDimensions。
3. 优先使用 Expo 官方或项目已验证的跨平台能力。
4. 共享业务逻辑、数据结构、路由语义和组件对外接口。
5. 不要因为当前主要使用浏览器调试，就把共享页面改成 HTML/CSS 页面。

（二）禁止未隔离的平台 API

共享代码中禁止直接使用且未进行平台隔离的：

- window
- document
- localStorage
- sessionStorage
- navigator 专属能力
- HTMLElement
- document.querySelector
- 浏览器专属文件 API
- 只适用于 Web 的事件对象
- 只适用于 Node.js 的模块
- 只支持 Android 或 iOS 的原生模块

确实需要 DOM API 时，必须满足以下一项：

1. 使用 Platform.OS === 'web' 进行完整隔离。
2. 放入 `.web.js` 平台专属文件。
3. 封装在明确的 Web 适配器中，并为 Android 和 iOS 提供对应实现或稳定降级。

（三）平台差异处理

平台差异按以下顺序处理：

1. 优先通过弹性布局和跨平台组件解决。
2. 其次使用 Platform.select。
3. 再使用 Platform.OS 条件分支。
4. 差异较大时使用 `.web.js`、`.native.js`、`.android.js` 或 `.ios.js`。
5. 平台专属文件必须保持一致的对外组件接口。
6. 不在页面业务代码中散落大量 Platform.OS 判断。
7. 可复用的差异封装到组件、Hook、工具函数或适配器中。

（四）平台专属能力必须提供降级

某项能力无法在 Web、Android 和 iOS 完全一致时，必须为其他平台提供稳定降级，不允许其中一端报错或丢失主要功能。

例如：

- Blur 不可用时使用半透明背景或渐变降级。
- Android 阴影使用 elevation 或项目已验证实现，Web 使用兼容阴影，iOS 使用支持的 shadow 属性。
- 原生分享不可用时，Web 提供复制或明确提示。
- 文件选择、键盘、富文本、权限和存储能力需要封装平台适配。
- 原生模块在 Web 不可用时必须提供 Web fallback。
- hover 只是 Web 增强，主要功能不能仅存在于 hover 状态。

（五）新依赖引入规则

新增依赖前必须同时检查：

1. 是否支持当前 Expo 版本。
2. 是否支持 React Native。
3. 是否支持 Expo Web。
4. 是否支持 Android。
5. 是否保持 iOS 代码兼容。
6. 是否需要修改原生工程。
7. 是否兼容 Expo 托管工作流。
8. 是否可以使用项目现有依赖完成。
9. 是否会导致 Web 构建失败。

如果新依赖只支持单一平台，必须说明必要性，进行平台隔离，并为其他平台提供替代实现。

（六）跨平台布局与交互

1. 使用 useWindowDimensions() 和弹性布局适配窄屏手机、常见 Android 屏幕、iPhone 逻辑尺寸和 Web 宽屏。
2. Web 宽屏可使用 maxWidth 并居中，但根据页面类型决定，不机械套用同一数值。
3. 主要交互同时考虑触摸、鼠标和键盘。
4. 共享代码使用 onPress，不直接依赖 Web onClick。
5. 移动端点击区域必须足够大。
6. 长文本需要允许换行，按钮和输入框不依赖固定文字长度。
7. 图片根据容器宽度等比缩放，不使用只适配单一屏幕的拉伸方式。
8. Web Safe Area 可以视为 0，Android 和 iOS 保留 useSafeAreaInsets() 逻辑。

（七）跨平台完成标准

页面必须满足：

1. Web 可构建、可打开、可交互。
2. Android 共享代码路径不存在明显不兼容实现。
3. iOS 共享代码路径不存在明显不兼容实现。
4. 三端路由语义和主要交互保持一致。
5. 未使用未隔离的 DOM API 或单平台原生模块。
6. 平台专属能力已提供降级。
7. 新依赖已完成 Expo Web 和 React Native 兼容性检查。
8. 图片、字体、Ionicons 和 FairySvgIcon 都使用跨平台加载方式。
9. 屏幕宽度变化不会造成明显遮挡、溢出或错位。
10. Web 通过但 Android 或 iOS 代码明显不兼容时，不能标记为完成。

没有 Android 或 iOS 设备时，不能声称已完成对应设备的实际视觉验证，但必须完成代码兼容性审查。

（八）每批跨平台检查

每完成 1–2 个页面，至少执行：

1. 在 Web 中实际打开目标路由。
2. 检查 Web 控制台和运行错误。
3. 检查窄屏和宽屏 Web 布局。
4. 执行项目约定的 Web 构建检查。
5. 搜索新增代码中未隔离的 window、document、localStorage、DOM API 和单平台模块。
6. 检查 Platform 分支是否都有有效返回或降级实现。
7. 检查新增依赖的 Web、Android 和 iOS 兼容性。
8. 检查图片、字体和图标加载方式。
9. 检查 Android 和 iOS Safe Area 逻辑是否仍然存在。
10. 在报告中记录尚未进行的 Android 和 iOS 设备级验证。

十、布局与平台规则

1. 优先使用 Flexbox 和正常文档流布局。
2. 使用 useWindowDimensions() 适配屏幕。
3. 不直接把设计图物理像素当作 React Native dp。
4. 不要求按照效果图坐标机械计算所有元素。
5. 避免大面积使用绝对定位。
6. 装饰性元素、浮层和确实需要固定位置的元素可以使用绝对定位。
7. 不使用大量绝对坐标来实现普通内容布局。
8. 优先使用项目主题 token。
9. 页面有特殊视觉需求时，允许使用合理的页面级样式。
10. 不绘制假的系统状态栏、时间、电量、信号或系统底部横条。
11. 修改页面前先检查 FairyPage 是否已经处理顶部 Safe Area。
12. 靠近底部的按钮、TabBar、列表和固定面板需要考虑 insets.bottom。
13. Web、Android 和 iOS 存在合理差异时，可以使用 Platform.select 或平台专属样式。
14. 不为了任何一个平台的视觉完全一致而破坏其他平台的运行和主要交互。
15. Web 和移动端兼容是项目开发的强制前提。Android 是最终重点验收平台，Web 是当前主要调试平台。任何一端出现构建失败、运行错误、路由不可用或主要交互缺失，都不能标记为完成；视觉表现允许根据平台特性存在合理差异。
16. 不通过禁止字体缩放来掩盖布局错误。
17. 图片必须保持原始宽高比例。
18. 文字不能明显溢出、截断或与图标重叠。

十一、图片、SVG 和 Icon 的使用边界

必须根据视觉元素的尺寸、用途和复杂度选择实现方式。

必须明确区分：

1. 小型功能图标
2. 简单规则图形
3. 复杂绘本插画和装饰图片

小型功能图标可以使用：

- @expo/vector-icons 中的 Ionicons
- 项目现有 FairySvgIcon 组件
- 简单、规模较小的 react-native-svg 图形

适合使用 Ionicons 或 FairySvgIcon 的元素包括：

- 返回
- 关闭
- 添加
- 删除
- 搜索
- 设置
- 分享
- 箭头
- 勾选
- 展开和收起
- 相机
- 图片
- 日历
- 五角星
- 四角星
- 爱心
- 小皇冠
- 小铃铛
- 小标签
- 小型功能性装饰符号
- 页面中显示尺寸较小的辅助图形

小图标优先检查 Ionicons 是否存在合适图标。

Ionicons 中存在合适图标时，可以直接使用，但需要调整：

- size
- color
- opacity
- stroke 的视觉重量
- 与文字和按钮的间距

如果 Ionicons 中没有足够美观或不符合项目气质的图标，可以使用 FairySvgIcon 绘制一个小型 SVG 图标。

FairySvgIcon 的使用条件：

1. 图标显示尺寸较小。
2. 图形结构相对简单。
3. SVG 路径数量可控。
4. 主要作用是功能提示或小型装饰。
5. 使用 SVG 能明显改善图标的美观程度。
6. 不会演变成一张复杂插画。

禁止使用 SVG 绘制大型复杂插画或草图。

以下内容不得为了省事而使用大量 SVG Path 勉强拼接：

- 人物
- 动物
- 复杂建筑
- 复杂礼盒
- 大型绘本场景
- 多物体组合插画
- 手绘日记本主体
- 复杂花束
- 带丰富纹理的贴纸
- 大面积纸张纹理
- 水彩场景
- 蜡笔质感主体
- 复杂光影
- 不规则柔边装饰
- 页面中的大型主视觉图片

如果一个视觉元素已经接近“插画”“贴纸”“场景”或“大型主视觉”，就不能使用 SVG 绘制草图代替。

SVG 和 Icon 主要服务于小图标，不负责承担页面大型视觉素材。

十二、复杂图片和图片生成模型

当页面需要大量绘制感图片、复杂插画、绘本贴纸或大型视觉素材时，必须使用当前环境提供的图片生成模型完成。

不能使用以下方式敷衍替代复杂图片：

- 使用大量 SVG Path 拼出草图
- 使用多个简单几何图形模拟复杂插画
- 使用 Emoji
- 使用不匹配的普通图标放大冒充插画
- 使用低质量占位图片
- 使用网络上风格不一致的相似图片
- 使用整张效果图作为页面背景
- 直接裁切效果图中的局部作为最终图片
- 将小图标强行放大成为主视觉

适合使用图片生成模型的内容包括：

- 童话绘本插画
- 人物和动物
- 复杂物品
- 大型礼盒、日记本、相册等主视觉
- 多物体组合场景
- 手绘贴纸
- 花束和复杂植物
- 纸张、蜡笔、水彩纹理
- 复杂背景装饰
- 带不规则柔边和光影的元素
- 具有明显绘画风格的视觉主体
- 页面中面积较大的装饰图片
- 无法通过一个简单小型 SVG 保持美观的图形

资源选择顺序：

1. 检查项目中是否已有风格合适、清晰度足够的本地图片。
2. 如果是小图标，检查 Ionicons。
3. 如果是需要定制美感的小图标，使用 FairySvgIcon。
4. 如果是简单规则形状，使用 View、StyleSheet 或渐变。
5. 如果是复杂图片、绘本插画或大型主视觉，使用图片生成模型。

效果图可以作为图片生成的重要视觉参考，用于理解：

- 主体类型
- 朝向
- 构图
- 色板
- 画风
- 视觉重量
- 在页面中的作用

图片生成不要求和效果图完全相同，但必须保持：

- 项目整体绘本风格
- 温暖柔和的配色
- 主体清晰
- 构图适合页面
- 与周围 UI 协调
- 不出现明显的异质画风

每个重要复杂素材应单独生成，不要把多个无关图标一次性生成在一张大图中。

生成后必须检查：

- 主体是否完整
- 构图是否适合页面
- 图片是否清晰
- 是否存在多余文字
- 是否存在水印
- 是否存在异常物体
- 是否与项目色板协调
- 透明背景是否正确
- 边缘是否干净
- 是否存在过大的无效透明区域
- 在页面实际显示尺寸下是否清晰
- 是否被拉伸或错误裁切

生成的最终素材应保存到项目正式 assets 目录，并通过稳定的 require/import 引入。

如果图片生成结果基本美观，只存在小范围问题，应优先使用图片编辑方式修正，不要每次完全重新生成并导致画风漂移。

图片资源治理规则：

1. 不手动批量移动、重命名或删除 assets 中的现有文件。
2. 资源整理前先扫描所有 require/import、资源注册表和页面引用。
3. 将资源分为：
   - 应用运行时正式资源
   - 设计参考图和效果图
   - 图片生成原始结果
   - 对比图、阶段性截图和构建产物
   - 已确认无用资源
4. 正式运行时资源统一放入 `assets/images`、`assets/fonts` 或明确的应用图标目录。
5. `assets/design`、`assets/design-assets` 和 `assets/page-effects` 默认视为设计参考区，不应成为新页面的直接运行时依赖。
6. 如果现有页面仍直接引用设计参考区资源，先将必需资源复制或迁移到正式运行时目录，更新所有引用并通过构建检查后，再处理原文件。
7. 移动或删除前必须再次扫描引用，不得依靠文件名猜测资源是否无用。
8. 设计参考资源不应意外进入正式构建。在运行时引用迁移完成后，可使用 `.easignore` 排除纯参考目录，但排除前必须确认没有运行时引用。
9. 图片生成原图与最终运行图分开管理。项目正式目录只保留页面实际使用的优化版。
10. 正式图片需要记录用途、生成提示词、最终路径、尺寸、透明通道和使用页面。
11. 统一使用简短、稳定、英文 kebab-case 文件名，避免将 `ChatGPT Image ...`、时间戳或含空格的临时文件名直接作为正式资源名。
12. 在保持视觉质量的前提下控制图片像素尺寸和文件体积，不将超出实际显示需求数倍的原始大图直接投入运行时。
13. 建立单一图片资源注册表，避免 `fairyImages.js` 与组件内部同时维护两套路径映射。
14. 只有在引用迁移、Web 构建和代码检查通过后，才能删除旧资源。

现有复杂 SVG 迁移规则：

1. 对 `FairyIllustration`、`FeaturePage` 以及页面中现有的大型 SVG 进行使用清单扫描。
2. 不一次性删除所有旧 SVG，避免破坏已有页面。
3. 当某个页面进入当前开发批次时，如果大型 SVG 承担主视觉、绘本插画或复杂贴纸作用，使用图片生成模型生成正式资源并替换。
4. 大型 SVG 可暂时作为开发 fallback，但不能作为完成页面的最终复杂主视觉。
5. 五角星、四角星、日历、箭头、爱心等小型符号继续使用 Ionicons 或 FairySvgIcon。

十三、没有效果图的页面

如果某个页面没有独立效果图，允许继续开发，不需要停止等待用户补图。

处理方式：

1. 参考同一业务模块已有页面。
2. 参考受保护页面的设计语言。
3. 参考项目主题 token。
4. 参考 Fairy 系列组件。
5. 根据页面用途设计合理的信息层级。
6. 保持与其他页面一致的字体、颜色、圆角、按钮和卡片风格。
7. 优先保证美观、完整和可使用。
8. 在阶段报告中注明该页面没有独立效果图，是根据项目设计体系完成的。

没有效果图不等于可以完全自由设计成其他 App 风格。

十四、代码实现规则

1. 优先使用项目已有主题 token。
2. 允许为页面增加合理的局部样式。
3. 优先复用项目现有组件。
4. 不为了复用组件而破坏页面美感。
5. 共享组件的新行为优先设计成可选属性。
6. 保留现有 Expo Router 路由。
7. 不修改与当前页面无关的业务逻辑。
8. 不覆盖用户未提交代码。
9. 不使用整张效果图作为背景伪装页面。
10. 普通文字必须使用 Text 实现。
11. 需要交互的控件必须可以点击和切换。
12. 图片使用稳定的 require/import。
13. 图片设置合理的 width、height 和 resizeMode。
14. 不拉伸图片改变原始比例。
15. 不留下 TODO 和临时占位内容。
16. 不大范围重构与当前页面无关的代码。
17. 修改共享组件前检查调用范围。
18. 发现用户已有实现可复用时优先沿用。
19. 开发辅助逻辑必须限制在开发环境。
20. 遇到明显错误时应直接修复，不只报告问题。

公共组件渐进治理规则：

1. `src/components` 下的全部现有组件都可以复用和渐进改进，不局限于 `FairyPage`、`FairyCard`、`FairyButton` 等示例组件。
2. 不在任务开始或恢复时集中重写全部公共组件；随页面批次检查、复用和完善，不因此重新执行全项目准备阶段。
3. 每个页面开发前，先检查是否已有匹配的公共组件、主题 token 或同一业务模块组件。
4. 同一种界面结构、交互或视觉模式在 2–3 个页面中重复出现时，评估提取为公共组件；确有复用价值时再实施。
5. 现有组件只缺少样式或行为能力时，优先增加向后兼容的可选属性、`variant`、插槽或组合能力，不直接改变默认表现。
6. 页面需求特殊且复用价值较低时，允许保留页面级组件和样式，不为了复用而破坏页面美感或制造过度抽象。
7. 新建公共组件前搜索 `src/components` 和相关业务目录，避免出现职责重复但名称不同的组件。
8. 修改 `FairyPage`、`FairyHeader`、`FairyButton`、`FairyCard`、`FairyTabBar`、`FairyImage`、`FairyInput` 等高使用率组件前检查调用范围，优先采用可选、向后兼容的实现。
9. 高使用率公共组件发生可见行为或布局变化后，使用 Chrome 抽查至少一个当前页面、一个既有调用页面；涉及受保护页面时，再抽查一个受保护页面。仅修改内部实现且确认输出不变时，可记录代码级影响检查而不强制重复完整视觉验收。
10. 在 `docs/ui-alignment-inventory.md` 中按需记录页面使用的主要公共组件、发现的重复模式和后续统一事项；不要求一次性补齐所有历史页面。
11. 每批报告增加“公共组件变更”项，记录本批复用、新增或扩展的组件、兼容方式、影响页面和抽查结果；没有变更时写明“无”。

页面骨架与顶部导航强制规则：

1. 所有具有实际可见内容的新页面，以及当前批次正在开发或明显调整结构的既有页面，默认必须以 `FairyPage` 作为页面根容器；页面主体内容必须完整作为 `FairyPage` 的 `children` 渲染，不在 `FairyPage` 外再放置页面级内容、背景或浮层。
2. 页面存在返回功能、顶部标题或右侧顶部操作中的任意一项时，必须使用 `FairyHeader`，并通过 `FairyPage` 的 `header` 属性传入：`<FairyPage header={<FairyHeader showBack title="页面标题" right={...} />}>...</FairyPage>`。
3. 禁止把 `FairyHeader` 直接放在 `FairyPage` 的滚动 `children` 顶部来模拟导航栏；这样不会启用 `FairyPage` 已有的顶部 Safe Area、固定头部、滚动模糊/阴影和正文避让逻辑。
4. 返回功能使用 `FairyHeader` 的 `showBack` 和可选 `backName`；标题使用 `title`；右侧按钮、图标或文字使用 `right`。不要在页面内重复手写返回按钮、居中标题和右侧操作组成的自定义顶部栏。
5. 当前 `FairyHeader` 只接收 `title`、`showBack`、`right`、`style`、`backName`。页面眉题、说明、副标题或大型视觉标题应放在 `FairyPage` 的主体 `children` 中，不能继续把未支持的 `eyebrow`、`subtitle` 等属性传给 `FairyHeader` 并假设其会显示。
6. 页面只有标题而没有返回功能时，仍使用 `FairyPage.header + FairyHeader`，保持 `showBack={false}`；页面只有返回功能而不需要标题时，使用 `FairyHeader showBack`，允许不传 `title`。
7. 纯重定向/别名路由、布局或 Provider 文件、无独立可见页面内容的包装组件不要求使用 `FairyPage`。登录、引导、主 Tab 根页、全屏沉浸式媒体页等确有设计依据的特殊页面可以不显示标准 `FairyHeader`，但仍应优先使用 `FairyPage` 处理页面根布局，并在批次报告中记录例外理由。
8. 受保护页面不因本规则单独发起批量迁移；当用户明确要求修改受保护页面，或当前改动实际触达其页面骨架时，再按影响范围进行兼容调整和回归。
9. 开发既有页面时，如果发现其将 `FairyHeader` 放在滚动内容中，或手写了等价顶部栏，应在当前页面批次内优先迁移到标准骨架；如果迁移存在明显行为风险，先记录差异和原因，不得静默保留后仍宣称满足本规则。
10. 使用标准头部后，正文顶部间距应通过 `FairyPage` 的 `topSpace`、`contentStyle` 等属性调整，不使用额外空白 View、重复 Safe Area padding 或硬编码状态栏高度补偿。
11. 页面验收必须检查：返回按钮可用、标题和右侧操作正确、正文未被头部遮挡、滚动后头部层级与阴影正常、Web 与移动端 Safe Area 合理、窄屏标题没有与两侧操作重叠。

模块引用与路径别名规则：

1. JavaScript 源码通过根目录 `jsconfig.json` 使用 Expo CLI 与 Metro 原生路径别名能力；当前只配置 `@/*` → `src/*`，不安装 `babel-plugin-module-resolver`，不为此修改 Babel 或 Metro 配置。
2. 新增文件或正在修改的文件，凡引用 `src` 下模块，默认使用 `@/components/...`、`@/theme/...`、`@/store/...`、`@/utils/...` 等别名。
3. 同目录或同一局部模块内的近距离引用继续使用 `./`；第三方包导入和 Expo Router 路由地址保持原样。
4. `@/` 只表示 `src/`。暂不配置 `@assets/*`，静态图片 `require` 与现有资源注册路径不做批量迁移。
5. 路径迁移跟随当前页面或当前业务模块的实际开发逐步完成；当前模块完成后，再迁移该模块剩余的 `../../src/...` 深层引用。
6. 不为了统一路径单独修改受保护页面，不对全项目约 440 处历史引用执行一次性机械替换。
7. 每批迁移后至少执行 Web 构建、Android Metro bundle/export 级检查、目标页面 Chrome 验证、`git diff --check`，并搜索目标文件确认不再残留本批计划迁移的深层 `src` 引用。
8. 如果任一平台无法解析别名，先回退该批别名试点修改并记录原因，不得牵连或回滚其他在途成果。

十五、页面实施流程

每批处理 1–2 个页面。

首次执行本计划时，先完成一次“阶段 0：稳定开发基线”。阶段 0 只执行一次，完成后再进入每批 1–2 页的自动开发循环。

阶段 0 必须完成：

1. 确认旧任务已经停止修改当前工作区。
2. 检查 Git 工作区，记录当前未提交成果，不回滚、不覆盖。
3. 创建或读取 `docs/resume-state.md`，将当前模块、批次、在途页面、工具状态和下一步入口与 Git、开发清单及实际代码核对；已有恢复记录不得未经核对直接覆盖。
4. 明确生产默认入口、Mock 入口、Web Dev UI 入口和首次启动入口，不将临时调试重定向当作正式产品入口。
5. 为 Web Dev UI 建立明确、可重复的启动方式，确保 `EXPO_PUBLIC_DEV_UI=1` 等开发变量只在开发环境生效。检查并复用现有 Expo Web 服务；失效时再启动新服务，记录实际 URL 和动态端口。
6. 确认当前任务已加载 `chrome:control-chrome` 技能，并使用 Chrome 打开或复用 Expo Web 标签页。Chrome 交互自测不得由普通命令行浏览器或独立 Playwright 代替。
7. 建立“正式路由—兼容别名—设计图—实现文件”清单。
8. 对 `/album` 与 `/photo/album`、评论、通知、AI progress 等重复或别名路由确定唯一正式实现；其他入口优先重定向，避免同一业务重复开发。
9. 检查 `fairyImages.js`、`FairyImage`、`FairyIllustration`、`FairyHeroImage` 和所有直接 require，建立单一正式图片资源注册表和大型 SVG 迁移清单。
10. 执行资源引用扫描和体积盘点，不立即批量移动 assets。
11. 明确 `src/data/mockData.js`、`src/api/mockData.js` 和 Zustand 持久化数据的职责。
12. 建立确定性 Mock 场景、数据版本和一键重置方式，避免 AsyncStorage 旧数据覆盖最新初始数据。
13. 检查 package.json 现有脚本，建立可重复的 Web 构建检查方式。如果项目缺少 lint、test 或 typecheck，如实记录，不伪造已执行的检查。
14. 对 `build-check-*`、`debug.log` 和其他生成产物建立清理与忽略规则，但修改 `.gitignore` 或删除前先确认文件确实无用。
15. 将 `FairyPage`、`FairyTabBar`、`AuthGate`、`app/_layout.js` 全局字体逻辑和资源注册表标记为高风险共享基础设施。
16. 修改高风险共享基础设施时，必须检查调用范围，优先使用可选属性和向后兼容实现。
17. 记录 `FairyTabBar` 与 `FairyPage tabSafe` 的底部 Safe Area 关系，使用移动端 inset 和 Web 0 inset 的跨平台方案，避免每页使用不一致的固定底部高度。
18. 检查全局字体在 Web、Android 和 iOS 的加载与图标字体保护，不因为单页字体问题随意修改全局注入逻辑。
19. 评估并建立路由级或全局错误边界，使单页运行错误不至于让整个调试流程白屏。
20. 完成一次 Web 构建和 Chrome 目标页面打开验证后，再进入首批页面开发，并把可恢复入口写入 `docs/resume-state.md`。

开始每批任务前：

1. 检查 git status。
2. 检查 git diff。
3. 阅读 `docs/resume-state.md` 和当前页面开发清单。
4. 根据恢复状态、Git 和实际代码确认是否存在上一批未完成工作；三者冲突时以实际代码和可运行验证为准，并修正恢复状态。
5. 找到对应效果图。
6. 找到路由和实现文件。
7. 检查主题、共享组件和已有图片资源。
8. 检查页面骨架：确认可见页面由 `FairyPage` 承载；存在返回、顶部标题或右侧操作时，确认 `FairyHeader` 通过 `FairyPage.header` 传入，而不是放在滚动 `children` 中或由页面手写等价顶部栏。
9. 判断页面中的视觉元素属于：
   - 普通文本
   - 小型 Icon
   - FairySvgIcon 小图标
   - 简单规则图形
   - 复杂图片或插画
10. 列出需要使用图片生成模型完成的复杂素材。
11. 给出简短实施说明，然后立即开始修改。

开发顺序：

1. 先保证页面结构和信息层级完整。
2. 先落实标准页面骨架：主体放入 `FairyPage` 的 `children`，返回/标题/顶部操作通过 `FairyPage.header + FairyHeader` 实现。
3. 完成主要交互。
4. 统一字体、颜色、间距、圆角、卡片和按钮。
5. 使用 Ionicons 或 FairySvgIcon 完成小图标。
6. 使用图片生成模型完成复杂插画和大型视觉素材。
7. 将正式素材接入页面。
8. 启动或复用 Expo Web 服务，并使用 `chrome:control-chrome` 在 Chrome 中运行页面。
9. 在 Chrome 中检查布局、滚动、文字、图片、交互和控制台问题，额外检查返回按钮、顶部标题、头部遮挡、滚动层级和窄屏两侧操作冲突。
10. 修复明显影响观感的问题。
11. 执行适合当前改动的检查。
12. 每完成一个页面、接入一张关键复杂图片或修复一个影响恢复的关键错误，立即更新 `docs/resume-state.md`，不要等待整批结束。
13. 完成 1–2 页后输出报告，并再次确认 `docs/resume-state.md` 已反映最新状态。
14. 如果不存在阻塞，自动进入下一批。

不要为了微小像素差异无限调整。

十六、页面完成标准

页面满足以下条件即可标记为“当前阶段完成”：

- 路由能够正常打开
- 页面运行无明显错误
- Web 和移动端共享代码路径可同时运行
- 不存在未隔离的 DOM API 或单平台原生依赖
- 平台专属能力已提供其他平台的降级实现
- 页面结构和内容完整
- 可见页面主体由 `FairyPage` 的 `children` 承载；存在返回、顶部标题或右侧操作时，通过 `FairyPage.header` 使用 `FairyHeader`，或已记录符合设计依据的例外
- 视觉方向与效果图协调
- 符合项目整体设计语言
- 页面整体美观
- 没有明显错位和粗糙区域
- 文字清晰，没有异常溢出或截断
- 图片没有拉伸、模糊或错误裁切
- 小图标大小、颜色和线宽协调
- 复杂主视觉没有使用 SVG 草图敷衍实现
- 需要复杂图片时已经使用图片生成模型
- 静态交互可以使用
- 滚动范围正常
- 系统状态栏没有被重复绘制
- 底部重要内容没有明显遮挡
- 常见 Web 预览宽度下布局正常
- Web 实际打开和构建检查通过
- Android 和 iOS 代码兼容性检查已完成
- 没有临时占位文字
- 没有无效素材
- 共享组件改动没有明显破坏其他页面
- 已执行与改动风险匹配的检查

如果尚未进行 Android 人工验证，标记为：

“页面开发完成，Web 已检查，Android 原生效果待人工验证。”

这不影响继续开发后续页面。

十七、任务优先级

按照以下顺序推进：

1. 继续完成当前已经开始但尚未收尾的页面。
2. 处理只有占位内容的页面。
3. 处理视觉结构明显不完整的页面。
4. 处理缺少主要插画或视觉素材的页面。
5. 处理已经基本完成、只需要细节优化的页面。
6. 每完成一个模块或约 6–8 页，整理一次 Android 人工验证清单。

如果当前 /empty-state 页面仍在处理中，先检查和完成它。

如果 /empty-state 已经完成，则根据 docs/ui-alignment-inventory.md 选择下一页。

十八、每批报告格式

每批完成 1–2 页后，输出简短报告：

- 本批完成页面
- 修改的代码文件
- 新增或修改的小图标
- 新增或生成的复杂图片素材
- 图片生成模型的使用情况
- 浏览器调试结果
- 执行的构建或代码检查
- Web、Android 和 iOS 兼容性检查结果
- 平台降级或尚未验证的平台能力
- 尚待 Android 验证的内容
- 已知问题
- 下一批准备处理的页面
- `docs/resume-state.md` 是否已经更新，以及其中记录的下一步准确入口

报告完成后，只要不存在以下情况，就自动继续下一批：

- 需要用户选择完全不同的设计方向
- 可能覆盖用户重要修改
- 必须修改受保护页面
- 需要扩大到当前项目之外
- 缺少完成任务所必需的文件
- 遇到无法安全处理的外部权限问题

不要因为一般性视觉判断、小范围样式选择或没有 Android 设备而暂停。

持久化恢复记录规则：

1. `docs/resume-state.md` 是中断恢复入口，不替代 Git、开发清单、阶段报告或实际运行验证。
2. 每完成一个页面、生成并接入关键图片、改变当前批次、修复阻塞错误或准备结束执行轮次时，立即更新恢复记录。
3. 恢复记录至少包含更新时间、当前模块、当前批次、已完成页面、在途页面、最近修改文件、图片路径、实际 Web URL、Chrome 验证路由、error、warning、未完成事项和下一步入口。
4. 系统重启或服务退出后，旧 URL 和工具状态只作为历史记录；新任务必须重新验证后更新，不能假设仍然有效。
5. 如果恢复记录与 Git、代码或 Chrome 实际表现冲突，以 Git、代码和实际验证为准，并立即修正恢复记录。
6. 不要等整批汇报时才第一次保存恢复状态。

十九、阶段性 Android 验证报告

每完成一个模块或约 6–8 页后，额外整理：

- 已完成页面路由
- 建议人工测试顺序
- 需要重点检查的 Safe Area
- 需要重点检查的 Android 阴影和 elevation
- 需要重点检查的字体差异
- 需要重点检查的底部导航区域
- 需要重点检查的图片清晰度
- 需要重点检查的键盘行为
- Web 中无法确认的问题
- 用户反馈 Android 问题时需要提供的截图和说明

输出该报告后，不等待用户反馈，立即继续下一批页面。

用户会在合适的时间手动进行 Android 调试。收到 Android 反馈后，将问题加入当前开发队列，在后续批次中进行原生端专项修正。

没有收到 Android 反馈时，继续按照页面开发清单推进，不暂停、不重复询问、不将任务标记为阻塞。

现在开始：

先完成第十五节中的“阶段 0：稳定开发基线”，再进入每批 1–2 页的开发循环。

检查 Git 工作区、现有未提交修改、`docs/resume-state.md`、docs/ui-alignment-inventory.md、当前正在处理的页面和相关效果图。

保留上一轮任务中可以利用的成果。

优先检查并完成当前在途页面。效果图作为重要视觉参考，但不要求 1:1 还原。以页面美观、符合项目设计语言、交互完整、代码稳定和浏览器可正常运行作为当前主要标准。

小型功能图标使用 Ionicons 或 FairySvgIcon。大型、复杂、具有绘本或手绘质感的图片必须使用图片生成模型完成，禁止使用大量 SVG Path 绘制草图代替复杂图片。

Web、Android 和 iOS 兼容是强制完成条件。Web 通过但移动端共享代码明显不兼容时，不能标记为完成。

每批完成 1–2 页，汇报后自动继续。

每完成一个模块或约 6–8 页，输出 Android 人工验证清单。输出后不要等待 Android 人工验证反馈，立即继续按照当前提示词和开发清单推进下一批。

Android 人工验证、设备连接和用户反馈都不是继续开发其他页面的前置条件。只有明确的原生端问题直接阻止相关页面继续实现时，才记录该局部问题；其余页面继续推进。
