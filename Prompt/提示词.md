二、最终验收平台

最终验收平台只考虑 Android。

Web 端可以用于快速预览、调试布局和检查基本样式，但 Web 与 Android 渲染不一致时，以 Android 真机或 Android 模拟器的实际效果为准。

为了让开发更方便，可以保留能够正常运行的 Web 代码，但不要为了让 Web 页面一致而牺牲 Android 效果。

如果某个视觉效果无法同时兼容 Android 和 Web，应使用 Platform.OS、Platform.select 或平台专属样式处理，并优先保证 Android。

Android 最终验收设备参数：

- 屏幕物理分辨率：1440 × 3168 px（竖屏）
- 设备标称分辨率：3168 × 1440（QHD+）
- 屏幕方向：竖屏
- 系统字体大小：标准
- App 字体缩放基准：fontScale = 1
- Android 是唯一最终视觉验收平台
- Web 只用于开发预览，不作为最终一致性标准

开始开发前，必须通过当前连接的 Android 真机或模拟器读取实际运行参数：

adb shell wm size
adb shell wm density
adb shell dumpsys window
adb exec-out screencap -p

必须区分：

1. 物理像素 px
2. React Native 逻辑尺寸 dp
3. PixelRatio
4. fontScale
5. 状态栏高度
6. 底部导航区或手势安全区高度
7. App 实际可用内容区域

禁止直接把 1440px 当作 React Native 页面宽度。

应在运行时读取：

PixelRatio.get()
PixelRatio.getFontScale()
useWindowDimensions()
useSafeAreaInsets()

页面尺寸计算以 useWindowDimensions() 返回的逻辑宽高为准；最终截图对比以 1440 × 3168 px 的 Android 原始截图为准。

如果 adb shell wm size 返回 Override size 或 adb shell wm density 返回 Override density，应以覆盖后的实际运行参数为准，并在验收报告中记录。

字体大小为系统“标准”，验收时不允许启用放大字体。页面普通字体不得通过禁止系统缩放来掩盖布局错误；但对于必须严格匹配效果图的装饰性文字，应明确记录并谨慎决定是否设置 allowFontScaling={false}。

十三、生成图片的画布和主体占比

最终图片必须按照“主体最大化、安全不裁切”的原则处理。

规则：

1. 主体应尽可能占满透明画布。
2. 主体完整可见，不能裁掉任何边缘。
3. 四周只保留防止抗锯齿、阴影、光晕被裁掉所需的最小安全留白。
4. 普通图标建议主体占画布宽高的 82%–92%。
5. 带阴影、光晕或柔边的插画可以保留更多安全留白。
6. 不允许主体只占画布中间很小一块。
7. 不允许存在大面积无意义透明区域。
8. 不允许非透明像素直接触碰画布四边。
9. 不使用自动裁切到贴边的 trim 结果。
10. 图片放入 React Native 时使用明确的 width、height 和合适的 resizeMode。
11. 不通过把一个低占比图片强行放大来弥补画布问题，应先修正图片画布。

注意：

“主体最大化”不等于“主体贴边”。

目标是在不裁切阴影、发光、柔边和抗锯齿像素的前提下，让主体尽可能填满画布。

十四、透明 PNG 生成规则

优先使用当前环境提供的内置图片生成能力。

如果工具不能直接生成可靠透明背景：

1. 使用纯色背景生成。
2. 背景必须是完全均匀的单一颜色。
3. 默认使用 #00FF00。
4. 如果主体中有绿色，改用 #FF00FF。
5. 背景不得包含：
   - 阴影
   - 渐变
   - 纹理
   - 地面
   - 反射
   - 光照变化
6. 主体不能使用抠图背景颜色。
7. 生成后在本地移除纯色背景。
8. 使用 soft matte 和 despill 处理抗锯齿边缘。
9. 验证输出包含 alpha 通道。
10. 检查四个角的 alpha 必须为 0。
11. 分别在白底、黑底和棋盘格背景预览。
12. 禁止出现绿色、紫色、白色或灰色边缘。
13. 禁止把仍带纯色底的图片投入项目。
14. 最终透明 PNG 必须保存到项目 assets 中，不能只留在临时目录或图片生成目录。

如果资源包含以下复杂透明内容，普通纯色抠图可能失败：

- 毛发
- 烟雾
- 玻璃
- 液体
- 半透明材质
- 反射
- 柔软光晕
- 复杂软阴影

遇到这种情况，应先向用户说明普通抠图的限制，再询问是否允许使用支持原生透明背景的图片生成路径。不得悄悄更换模型或生成方式。

- 不得使用背景颜色作为主体颜色

Avoid:
相似但不同的图标、图标库风格、Emoji 风格、赛博科技感、写实摄影、3D 塑料感、金属质感、过度光泽、高饱和荧光色、不同画风、额外装饰、主体过小、大面积空白、主体裁切、白边、灰边、纯色脏边、文字、水印

十六、生成结果验收

图片生成后，不能直接使用。

必须检查：

- 是否只包含目标主体
- 轮廓是否接近参考图
- 角度是否一致
- 比例是否一致
- 风格是否一致
- 色板是否一致
- 是否添加了多余元素
- 主体是否过小
- 透明区域是否过大
- 主体是否被裁切
- 阴影和抗锯齿是否完整
- 是否存在背景残留
- 四角是否透明
- 是否存在白边、灰边、绿边或紫边
- 页面实际显示尺寸是否清晰
- Android 真机/模拟器上是否出现模糊

如果生成结果不符合要求，必须进行单一目标迭代，例如：

- 只修正主体角度；
- 只修正丝带形状；
- 只减少透明留白；
- 只降低粉色饱和度；
- 只清理绿色边缘。

不要每次完全改写提示词，避免其他已正确特征发生漂移。

十七、代码实现规则

1. 优先使用项目已有主题 token。
2. 但现有 token 与效果图不一致时，允许为页面增加局部精确值。
3. 优先复用 Fairy 系列组件。
4. 但不得为了复用组件而改变效果图。
5. 共享组件的新视觉行为优先设计成可选属性，避免影响黄金基准页面。
6. 保留现有 Expo Router 路由。
7. 不修改与当前页面无关的业务逻辑。
8. 不覆盖用户未提交的代码。
9. 不使用整张页面效果图作为背景伪装页面。
10. 普通文字必须使用 Text 实现。
11. 需要静态交互的控件必须可以点击和切换。
12. 页面使用模拟数据时，数据内容应与效果图一致。
13. Android 阴影应使用 Android 可用的实现验证，不能只看 Web CSS 阴影。
14. Android 和 Web 效果冲突时，Android 优先。
15. 图片必须使用稳定的本地 require/import。
16. 图片尺寸和 resizeMode 必须明确。
17. 不允许通过拉伸图片改变原图比例。
18. 不允许留下临时占位图和 TODO 文字。

十八、页面实施流程

每次处理一个页面或一组紧密相关页面。

固定流程：

1. 找到对应效果图。
2. 找到对应路由和实现文件。
3. 读取效果图真实宽高。
4. 确认 Android 验收设备尺寸。
5. 分析页面图层。
6. 建立视觉测量记录。
7. 列出需要代码实现的元素。
8. 列出需要重新生成的图片资源。
9. 先生成并验收缺失资源。
10. 再实现页面代码。
11. 在 Web 上进行可选的快速预览。
12. 在 Android 模拟器或真机上正式运行。
13. 截取 Android 页面。
14. 将 Android 截图与效果图统一到同一内容尺寸。
15. 生成叠图、并排对比图或差异图。
16. 修正最大的视觉差异。
17. 重新运行和截图。
18. 通过后再处理下一个页面。

十九、视觉对比顺序

优先修复：

1. 整体缩放
2. Safe Area
3. 页面整体上下偏移
4. 页面左右边距
5. 卡片坐标和宽高
6. 图层顺序
7. 图片尺寸和位置
8. 图片主体占比
9. 字体大小和行高
10. 模块间距

然后修复：

- 阴影
- 渐变
- 描边透明度
- 圆角
- 字间距
- 图标细节
- 装饰元素
- 背景纹理

不能只通过阅读代码宣布视觉完成。

二十、完成标准

页面只有同时满足以下条件才能标记为完成：

- 对应效果图明确
- Android 路由能够打开
- Android 运行无错误
- 页面没有临时占位内容
- 文字是可编辑文本
- 静态交互可以使用
- 系统状态栏没有被重复绘制
- 底部内容没有被系统导航区域遮挡
- 主要坐标和尺寸与效果图对齐
- 图片资源来源明确
- 没有直接使用效果图裁片作为最终资源
- 没有使用风格不一致的相似图标
- 生成图片具有正确透明通道
- 生成图片主体占比合理
- 图片无大面积无效透明区域
- 图片没有拉伸、裁切、脏边和背景残留
- 已截取 Android 实际运行截图
- 已与效果图对比
- 共享组件修改没有破坏黄金基准页面
- 无法消除的差异已经说明原因

1. 建立完整页面清单。

   图片资源 → 页面代码 → Android 运行 → 截图 → 对比 → 修正。

- 已完成页面

- 使用的图片生成提示词
- Android 验收设备
- Android 对比截图路径
- 黄金基准页面回归结果
- 已知差异
- 下一批页面
